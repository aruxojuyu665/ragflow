# Руководство по внесению вкладов

Общие рекомендации для участников сообщества RAGFlow.

---

В этом документе приведены рекомендации и основные моменты, которые следует учитывать при отправке ваших вкладов в RAGFlow.

- Чтобы сообщить об ошибке, создайте [GitHub issue](https://github.com/infiniflow/ragflow/issues/new/choose).
- Для дополнительных вопросов вы можете изучить существующие обсуждения или начать новое в разделе [Discussions](https://github.com/orgs/infiniflow/discussions).

## Что вы можете внести

Ниже приведён список возможных вкладов, но он не является исчерпывающим.

- Предложение или реализация новых функций
- Исправление ошибок
- Добавление тестовых случаев или демонстраций
- Публикация блога или руководства
- Обновления существующей документации, кода или аннотаций
- Предложение более удобных для пользователя кодов ошибок

## Создание pull request (PR)

### Общий рабочий процесс

1. Форкните наш репозиторий на GitHub.
2. Клонируйте ваш форк на локальную машину:
`git clone git@github.com:<yourname>/ragflow.git`
3. Создайте локальную ветку:
`git checkout -b my-branch`
4. Укажите достаточную информацию в сообщении коммита
`git commit -m 'Provide sufficient info in your commit message'`
5. Зафиксируйте изменения в локальной ветке и отправьте на GitHub: (включите необходимое сообщение коммита)
`git push origin my-branch.`
6. Отправьте pull request на рассмотрение.

### Перед созданием PR

- Рассмотрите возможность разделения большого PR на несколько меньших, автономных PR для поддержания отслеживаемой истории разработки.
- Убедитесь, что ваш PR решает только одну задачу, или оставляйте несвязанные изменения минимальными.
- Добавляйте тестовые случаи при внесении новых функций. Они демонстрируют корректную работу вашего кода и защищают от потенциальных проблем при будущих изменениях.

### Описание вашего PR

- Убедитесь, что заголовок PR краткий и понятный, содержащий всю необходимую информацию.
- При необходимости укажите соответствующий GitHub issue в описании PR.
- Включите достаточные детали дизайна для *критических изменений* или *изменений API* в описании.

### Проверка и слияние PR

Убедитесь, что ваш PR проходит все тесты Continuous Integration (CI) перед слиянием.
